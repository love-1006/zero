## 이번 세션에서 만든 것

### login-service (:8000) — 기존 서비스 보강
- 인증/인가 성공·실패 이벤트 로깅 (`admin_auth.py`, `auth.py`, `user.py`) — 사용자 입력 필드는 `%r`로 로깅해서 로그인젝션 방지
- `/administrator-login`, `/administrator-signup`에 in-memory rate limiting (`app/services/rate_limiter.py`) — IP당/login_id당 슬라이딩 윈도우
- Google/Apple OAuth 라우트 스켈레톤 추가 (`app/services/oauth/google.py`, `apple.py`) — Google은 완전 구현(자격증명만 없음), Apple은 `build_authorize_url`만 구현하고 토큰교환/프로필조회는 명시적으로 `NotImplementedError` 성격의 `OAuthExchangeError` (Team ID/Key ID/Private Key로 서명하는 client assertion JWT가 필요해서 자격증명 없이는 정확히 구현 불가 — 틀린 코드보다 명시적 미구현이 낫다고 판단)
- **버그 수정**: `turnstile.py`가 Cloudflare `siteverify`의 non-200 응답(예: 잘못된 secret key)을 처리 못 해서 캡차 실패가 500으로 새던 것을 401로 고침 — 실제 로컬 DB로 재현/검증함
- `main.py` 로그 타임스탬프 UTC 명시, 전역 예외 핸들러(스택트레이스 미노출) 추가

### admin-service (:8008) — 신규
- 관리자 인증 게이트웨이 (BFF). 자체 테이블 없음.
- `get_current_admin`: JWT 검증(HS256 allowlist) + `role=="admin"` 확인, `Authorization: Bearer` 헤더 방식
- `/health`, `/admin/me`만 구현. AD-0101 등 실제 프록시 엔드포인트는 Product 등 대상 서비스가 아직 없어서 스킵.

### main-service (:8010) — 신규
- 홈 화면 BFF. **자체 소유 테이블 있음**: `service.user_health_profiles`, `service.user_preferences` (2026-07-15 팀 결정으로 Main 소유 확정 — `backend/services/README.md`, `main-service.md` 갱신 완료)
- 인증은 `usr={JWT_TOKEN}` 쿼리/바디 파라미터 방식 (기능명세서 API-Spec, login-service와 통일 — admin-service의 Bearer 헤더 방식과 다름, 의도적 결정)
- 구현: `GET/PUT /home/health-profile`(동의 게이팅, DB CHECK 제약을 애플리케이션에서 먼저 검증), `GET/POST/DELETE /home/preferences`(태그기반 vs 커스텀텍스트 배타조건, IDOR 방지), `GET /home/user-sugar-calorie`(KST 기준 오늘 섭취량 — `meal_logs`+`v_meal_totals` 조인, Diet 소유 데이터 읽기전용), `GET /home/user-recommend`(관심카테고리 기반 상품 추천 — `products`+`product_tags` 조인, Product 소유 데이터 읽기전용)
- **플레이스홀더(스탠바이)**: `GET /search`(MN-0102), `GET /home/rank/item`(MN-0110) — `{"status":"PREPARING", ...}` 반환. 검색은 ElasticSearch, 랭킹은 Kafka→MongoDB 분석 파이프라인 필요한데 둘 다 미연동. 파라미터/응답 스키마는 기능명세서와 이미 맞춰뒀어서 데이터 연동되면 `return`문만 교체하면 됨.
- **의도적으로 뺀 것** (사용자 지시): NT-0101(신제품 알림 — 푸시는 별도 시스템, 웹 알림함은 나중), MN-0111~0113(AI 챗봇/OCR — 프론트가 완성되면 직접 연결 예정)
- `create_all()` 안 씀 — `service` 스키마는 데이터팀 관리 대상이라 이 서비스가 소유한 두 테이블만 별도 로직으로 생성(주의: main-service는 로그인서비스처럼 자체 lifespan에서 `Base.metadata.create_all()`을 쓰되 OWNED 테이블만 대상으로 해야 함 — 실제 코드에서 `service.tags`/`Product`/`MealLog` 등 읽기전용 모델은 절대 create_all 대상 아님)

### community-service (:8012) — 신규
- 공지사항 + 좋아요. **자체 `community` 스키마를 스스로 생성/관리** (`CREATE SCHEMA IF NOT EXISTS community` + `create_all(tables=OWNED_TABLES)`만, `service` 스키마는 안 건드림)
- `notices`/`notice_likes`가 `public.users(id)`에 **진짜 FK** — 단, FK 해석을 위해 `app/models/user_ref.py`에 최소 stub User 모델을 등록해야 했음(안 그러면 SQLAlchemy `create_all()`이 `NoReferencedTableError`로 죽음 — 실제로 겪은 삽질, 새 서비스 만들 때 참고)
- 쓰기(작성/수정/삭제)는 `get_current_admin_from_token`으로 관리자 전용, 조회/좋아요는 공개 또는 일반 로그인 유저
- 감미료 정보(CM-0107/0108)는 자체 테이블 없이 Ingredients 소유 `service.tags`(`tag_type='SWEETENER'`) 읽기전용 재사용
- 기능명세서상 "권한체크 + 실제쓰기가 한 엔드포인트에 섞인" 설계는 깨끗한 REST(GET/POST/PUT/DELETE)로 재설계함 (사용자 승인받음)

### recipe-service (:8014) — 신규, **아직 커밋 안 됨**
- **완전 읽기전용, 자체 테이블 전혀 없음.** `recipes`/`recipe_ingredients`/`recipe_base_comparisons`/`recipe_ingredient_products`/`raw_ingredient_nutrients`는 전부 데이터팀의 "저당 레시피 유튜브 파이프라인"이 소유·적재 (`/Volumes/minoharu/douzone/final_team-alpha/레시피_데이터명세서_v0.2.xlsx` 참고).
- **주의**: `backend/services/recipe-service.md` 문서는 "자체 DB 필요"라고 되어 있는데 **최신화가 안 된 상태**임. 실제로는 이미 `service` 스키마에 테이블이 다 있고 FK도 걸려있어서 그냥 읽기전용으로 조인하면 됨. 문서 자체는 아직 안 고쳤음 — 필요하면 갱신할 것.
- 실제 원격 DB(211.46.52.151:15432/test_db) 조회 결과 **5개 테이블 전부 0건**(xlsx 문서의 "1,697건 적재" 등은 다른 환경 스냅샷으로 추정). 스키마/FK는 실제와 정확히 맞춰서 만들었으므로 데이터 적재되면 코드 변경 없이 바로 동작.
- `GET /recipes`, `GET /recipes/{id}`(재료+당/칼로리 비교), `GET /recipes/{id}/substitutes`(AI 대체상품 — `recipe_ingredient_products`의 pgvector 매칭 결과를 그대로 읽음, 실시간 계산 없음), `GET /recipes/{id}/exists`(Diet/Main의 느슨한 참조 검증용)
- **"억지 매핑 금지" 원칙 적용**: `recipe_base_comparisons.base_product_id`는 실제 FK가 없고 `base_products` 테이블 자체가 없어서(컬럼 미정) 원시 id값만 노출, 조인 안 함.

## DB 관련 핵심 사실 (다음 서비스 만들 때 재확인 필요 없게 정리)

- 실제 DB: `211.46.52.151:15432/test_db` (login-service `.env`에 자격증명 있음, `.env.example`엔 플레이스홀더만)
- 스키마 2개: `public`(User/Auth, login-service 소유, `users`/`social_accounts`/`admin_accounts`) / `service`(나머지 7개 서비스 도메인 데이터, **데이터팀이 관리·마이그레이션**)
- **`service` 스키마는 절대 DDL 금지** — 각 서비스는 read-only 모델만 만들고 `create_all()` 대상에서 제외해야 함(`OWNED_TABLES`처럼 명시적으로 리스트 분리하는 패턴 사용 중)
- `service` 스키마 안에 이미 존재 확인한 테이블: `products`, `product_tags`, `tags`, `meal_logs`, `meal_items`, `v_meal_totals`(view), `user_favorites`, `user_health_profiles`, `user_preferences`, `recipes`, `recipe_ingredients`, `recipe_base_comparisons`, `recipe_ingredient_products`, `raw_ingredient_nutrients` — 컬럼까지 다 확인한 건 위 4개 서비스 코드의 모델 파일 참고하면 됨(추측 아니고 실제 DB에서 `\d`로 뽑은 것)
- `base_products` 테이블은 아직 없음(컬럼 미정, recipe_base_comparisons.base_product_id가 느슨하게 참조만 함)
- 새 서비스가 `public.users`에 FK를 걸어야 하면 community-service의 `user_ref.py` 패턴(최소 stub 모델, OWNED_TABLES에서 제외) 참고할 것

## 공통 컨벤션

- 인증 토큰: **`usr={JWT_TOKEN}`을 쿼리/바디 파라미터로** (GET은 쿼리, POST/PUT은 바디 필드) — main/community-service가 이 방식. admin-service만 예외로 `Authorization: Bearer` 헤더(먼저 만들어서 이렇게 됨, 완전히 통일하려면 admin-service 방식 변경도 고려 가능)
- 관리자 권한: JWT의 `role` 클레임이 `"admin"`인지 확인 (모든 서비스가 같은 `JWT_SECRET` 공유, login-service가 발급)
- 에러 응답: 전역 예외 핸들러로 스택트레이스 미노출 + 일관된 `{"detail": "..."}` 형식
- 로깅: UTC 타임스탬프, 사용자 입력 필드는 `%r`(로그인젝션 방지), 인증/인가 성공·실패 이벤트 필수 로깅
- 검증 방식: 매번 로컬 Docker에 실제 스키마와 동일한 구조로 임시 Postgres를 띄워서 검증 (공유 원격 DB는 **읽기전용 조회만**, 쓰기 테스트는 항상 로컬 디스포저블 DB 사용 — 원격 DB 자격증명으로 아무 작업이나 하기 전엔 매번 사용자 승인 받음)
