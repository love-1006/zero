# 프론트 연동 전 확인 요청 — API 명세서 대비 변경/차이점

> 작성일: 2026-07-16
> 대상 문서: `기능명세서.xlsx` (진짜기능명세 / 진짜API-Spec / 제로파인더 기능명세 시트)
> 대상 서비스: login, admin, main, product, ingredients, diet, community, recipe (총 8개)

프론트 연동 시작 전에 팀장님/프론트에 꼭 공유해야 할 사항을 정리했습니다. 백엔드 코드와 최신 명세서를 실제로 대조해서 나온 **진짜 차이점**만 담았고, 이미 구두로 합의된 사항(중복 구현 허용 등)은 제외했습니다.

---

## 0. `{DOMAIN}/b/{경로}` 프리픽스 — 확인 완료

API-Spec 시트의 "Backend Path"는 `{DOMAIN}/b/{경로}` 형태로 붙는데, **게이트웨이에서 `/b`를 붙여주는 구조로 이미 잡혀 있습니다.** 프론트 담당이 `.env`로 유연하게 설정할 예정이라, 각 서비스 코드(`/home/search`, `/admin` 등 `/b` 없는 경로 그대로)는 수정할 필요 없습니다. 별도 조치 불필요.

---

## 1. 명세서 자체의 정합성 문제 (RC-01xx 번호 꼬임)

`진짜기능명세` 시트와 `진짜API-Spec` 시트에서 RC-0104 이후 번호가 서로 어긋나 있습니다.

| ID | 기능명세 시트 | API-Spec 시트 |
| --- | --- | --- |
| RC-0104 | 제품 사진 분석 (OCR) | `/diet/other-foods` (저장된 분석 결과 조회) |
| RC-0105 | 다른 제품 추천 (대체 상품) | `/diet/calender` (캘린더) |
| RC-0106 | 캘린더 | `/receipe/low-sugar` (저당 레시피 목록) |

API-Spec 시트에는 RC-0103(`/diet/ai-analyze`)과 RC-0104(`/diet/other-foods`) 사이에 **ID가 없는 새 행** `/diet/ocr-analyze`가 끼어 있는데, 이게 기능명세 시트의 RC-0104(OCR 분석)에 해당하는 것으로 보입니다. 즉 API-Spec 시트에 새 행을 추가하면서 뒤 번호를 안 밀어서 두 시트가 어긋난 상태입니다.

저희 diet-service 코드 내부 주석(RC-ID 라벨)은 **예전 번호 체계**(RC-0105=대체상품, RC-0106=캘린더)로 달려 있어서 지금 API-Spec 시트 번호와도 안 맞습니다. **실제 URL 경로 자체는 문제없이 동작**하니 급한 버그는 아니지만, 명세서 ID 정리가 먼저 되어야 서로 헷갈리지 않을 것 같습니다.

---

## 2. 명세에는 있는데 아직 구현 안 된 것

| 기능 | 명세 상 경로 | 상태 |
| --- | --- | --- |
| MN-0112 사진 기반 검색(OCR/렌즈) | `/search/lens` | **미구현.** 어떤 서비스도 이미지 업로드 기반 상품 검색을 갖고 있지 않습니다. |
| 식단 OCR 분석 (RC-0104, 기능명세 기준) | `/diet/ocr-analyze` | **미구현.** diet-service엔 `/diet/ai-analyze`(Vision AI, PREPARING 상태)만 있고 OCR 전용 엔드포인트는 없습니다. |
| 저당 레시피 전용 목록 (RC-0106) | `/receipe/low-sugar` | **미구현.** recipe-service엔 일반 목록(`GET /recipes`)만 있고, 저당 필터/정렬 전용 엔드포인트는 없습니다. |
| 레시피 AI 저당 비교 요약 (RC-0110, 신규 행 `/receipe/{n}/ai`) | `/receipe/{n}/ai` | **미구현.** 비교 데이터(`baseComparison`) 자체는 `GET /recipes/{id}` 응답에 이미 포함돼 있지만, 별도 AI 요약 엔드포인트는 없습니다. |
| US-0108 마이페이지 수정 | `/user/mypage` (`edit=True`) | **전용 엔드포인트 없음.** 필드 수정은 `POST /user/firstset` 재호출로만 가능합니다. 프론트에 이 부분 명확히 안내 필요합니다. |

## 3. 구현은 됐는데 명세서엔 아직 없는 것 (경로 재확인 필요)

- **레시피 대체상품 추천** (기능명세 RC-0109, "기존 레시피 재료에서 대체할 수 있는 제로/저당 제품 추천")은 실제로 구현돼 있습니다(`GET /recipes/{id}/substitutes`). 다만 API-Spec 시트엔 이 기능의 URL 행 자체가 없어서, 프론트가 참조할 공식 계약이 없는 상태입니다 → 시트에 행 추가 요청 필요.
- recipe-service 경로는 `/recipes/*` (복수, 정상 철자)인데, 명세서엔 `/receipe/*` (오타, 단수)로 적혀 있습니다. 오타를 명세서에서 고칠지, 저희가 오타 경로로 맞출지 결정이 필요합니다 (권장: 명세서 오타 수정).
- main-service에 `/home/health-profile`(GET/PUT), `/home/preferences`(GET/POST/DELETE), `/home/me`(GET)가 있는데, 이건 API-Spec 시트엔 없는 엔드포인트입니다. 이전에 "로그인 유저 맞춤 정보 제공"을 메인 서비스에 두기로 직접 지시하셨던 부분이라 의도된 추가이지만, 명세서에도 반영해두는 게 좋을 것 같습니다.

## 4. 참고 — 의도된 중복 (이미 구두 확인됨, 재론 불필요)

- `GET /search`: main-service(PREPARING placeholder)와 product-service(실제 구현) 둘 다 보유. **게이트웨이 라우팅 시 실제 트래픽은 product-service로 가야 합니다** — main-service 쪽으로 가면 빈 결과만 나옵니다.
- 인기 랭킹(`/home/rank/item`, MN-0110)과 감미료 정보(main-service 계열 vs ingredients-service `/community/gam-list`)도 기존에 안내드린 대로 중복 허용된 부분이라 그대로 유지합니다.

## 5. 별도 논의 필요 — "제로파인더 기능명세" 시트

명세서에 새로 추가된 시트로, SR-/PD-/SW-/RC-/ST-/EV- 코드 체계를 쓰는 완전히 별도의 기능 목록입니다 (OCR 스캔 파이프라인, 매장 연동, Kafka 이벤트/모니터링 인프라 등). 저희가 만든 8개 서비스 중 어디에도 아직 배정되지 않았고, 상당수(EV-02xx 등)는 백엔드보다 데이터/인프라팀 영역으로 보입니다. 어느 서비스가 담당할지 팀장님 확인 부탁드립니다.

---

### 요약 (팀장님께 전달용 3줄)

1. `/b/` 프리픽스는 게이트웨이가 처리, 프론트가 `.env`로 관리 — 백엔드 조치 불필요.
2. API-Spec 시트 RC-01xx 번호가 기능명세 시트와 어긋나 있어 정리 필요 (실제 URL은 정상 동작).
3. OCR 검색(`/search/lens`), 식단 OCR 분석(`/diet/ocr-analyze`), 저당 레시피 목록(`/receipe/low-sugar`), 레시피 AI 비교 요약(`/receipe/{n}/ai`)은 명세엔 있지만 아직 미구현 — 우선순위 정해주시면 진행하겠습니다.
