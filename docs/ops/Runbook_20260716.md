## 0. 이 문서를 왜 이런 구조로 만들었나

Google SRE Book은 "Effective Troubleshooting" 챕터에서 트러블슈팅을 **"문제 인지 → 진단 →
처방(treatment)"**의 반복 루프로 설명한다. 알림이 울렸을 때 사람이 매번 이 루프를 처음부터
다시 생각하면 느리고 실수하기 쉽다. Runbook의 역할은 "진단" 단계를 미리 표준화해서, 알림이
오면 **생각하지 않고 순서대로 확인**할 수 있게 만드는 것이다.

그래서 이 문서는 알림 규칙 하나당 정확히 같은 4개 항목(증상/원인 후보/확인 명령/조치)으로
통일했다. 알림 이름만 보고 이 표에서 해당 행을 찾아 그대로 따라가면 된다.

## 1. 알림별 대응 절차

### 1-1. High CPU Usage (경고, 85% 10분 지속)
- **원인 후보**: 정상적인 부하 증가(CI 빌드 등) vs 무한루프/좀비 프로세스
- **확인**: `ssh` 접속 후 `top`/`htop`으로 어느 프로세스가 CPU를 먹는지 확인. harbor VM이면
  현재 CI 빌드가 돌고 있는지 GitHub Actions 러너 상태부터 확인(CI 빌드 중이면 정상, 지속시간
  10분으로 이미 CI 스파이크는 걸러지도록 조정돼 있음 — 그런데도 울렸다면 CI 때문이 아닐 가능성)
- **조치**: 원인 프로세스가 CI가 아니면 어떤 프로세스인지 파악 후 필요 시 종료. 반복되면
  VM 스펙 증설 검토(단, 현재 리소스 여유가 빠듯하므로 팀 논의 필요)

### 1-2. High Memory Usage (경고, 90% 5분 지속)
- **원인 후보**: 메모리 누수, 예상보다 큰 워크로드
- **확인**: `free -h`, `docker stats`로 컨테이너별 메모리 사용량 확인
- **조치**: 특정 컨테이너가 계속 늘어나기만 하면(재시작해도 반복되면) 누수 의심 — 해당
  서비스 담당자에게 공유. 임시 완화로 컨테이너 재시작 가능하나 근본 원인 파악 전 반복 재시작만
  하지 말 것(SRE Book이 경고하는 "증상만 없애는 조치"의 전형적 예)

### 1-3. High Disk Usage (경고, 85%) / Critical Disk Usage (위험, 95%)
- **원인 후보**: 로그 누적, Docker 이미지/컨테이너 잔해, Prometheus/Loki 데이터 증가
- **확인**: `df -h`로 파티션 확인 → `du -sh /* 2>/dev/null | sort -rh | head`으로 큰 디렉터리
  탐색. Docker라면 `docker system df`
- **조치**: `docker system prune`(사용 안 하는 이미지/컨테이너 정리, **단 운영 중인 컨테이너에
  영향 없는지 먼저 확인**), 오래된 로그 정리. Critical(95%)은 디스크 풀로 인한 서비스 전체
  장애 임박 상태이므로 경고(85%)보다 훨씬 빠르게 대응해야 한다

### 1-4. Instance Down (위험, 1분 지속)
- **원인 후보**: VM 자체 다운, node_exporter 프로세스 죽음, 네트워크(Tailscale) 단절
- **확인**: 먼저 Tailscale 연결 상태 확인(`tailscale status` 또는 다른 경로로 ping) — 네트워크
  문제인지 VM 자체 문제인지부터 구분. VM이 살아있다면 `systemctl status node_exporter` 확인
- **조치**: node_exporter만 죽었으면 `sudo systemctl restart node_exporter`. VM 자체가
  안 뜨면 Proxmox 콘솔에서 상태 확인 후 재부팅. **재부팅 후에는 반드시 Harbor라면
  `systemctl status harbor`까지 확인**(과거 재부팅 시 자동기동 안 되던 사고가 있었음 —
  `SESSION_HANDOFF_harbor_push.md` 5-5 참고, 지금은 systemd 유닛으로 고쳐져 있어야 정상)

### 1-5. Blackbox Probe Failed (위험, 2분 지속)
- **원인 후보**: 대상 서비스(Harbor/Grafana/SonarQube/ci_sandbox) 자체 다운, 방화벽/네트워크 변경, blackbox-exporter 자체 문제
- **확인**: 어떤 `instance` 라벨이 실패했는지 Grafana 알림 상세에서 확인 → 해당 서비스에
  직접 curl로 접속 시도(`curl -v <해당 URL>`)해서 실제로 안 되는지, blackbox-exporter만
  이상한지 구분
- **조치**: 대상 서비스가 실제로 다운이면 그 서비스의 개별 런북(위 항목들)으로. blackbox-exporter
  자체 문제면 `docker compose restart blackbox-exporter`. **job=blackbox-http-ci-dependency
  (SonarQube)는 CI 파이프라인만 영향받고 사용자 서비스는 안 죽는다는 걸 먼저 확인** —
  성격이 다른 두 종류를 같은 알림으로 받는 상태이니 대응 우선순위를 헷갈리지 말 것

### 1-6. High App Error Rate (위험, 5xx 비율 5% 초과 5분 지속)
- **원인 후보**: 앱 코드 버그, 의존 서비스(DB 등) 장애, 배포 직후 회귀
- **확인**: Grafana Loki에서 `ci_sandbox` 로그 중 5xx 관련 에러 메시지 확인, 최근 배포
  이력(`gh run list`) 확인 — 최근 배포 직후라면 배포가 원인일 가능성부터 검토
- **조치**: 최근 배포가 원인이면 `scripts/rollback.sh`로 직전 정상 버전으로 롤백(자동 롤백은
  헬스체크 실패 시에만 동작하므로, 배포는 성공했지만 5xx가 나는 경우엔 수동 롤백 필요).
  배포와 무관하면 로그 기반으로 근본 원인 추적

### 1-7. TLS Certificate Expiring Soon (경고, 14일 미만) / Expiring Imminently (위험, 3일 미만)
- **원인 후보**: 인증서 갱신을 깜빡함
- **확인**: 어떤 대상의 인증서인지(현재는 Harbor만 해당) 확인
- **조치**: Harbor는 자체 서명 인증서를 쓰고 있어 만료가 임박하면 Harbor의 인증서 재발급
  절차(`harbor.yml`의 `https` 섹션, 공식 문서 "Configure HTTPS Access to Harbor" 참고)를
  따라 재발급 후 `docker compose down && docker compose up -d`로 재적용. **Critical(3일
  미만)까지 갔다면 재발급 직후 Blackbox 프로브가 다시 정상(`probe_success: 1`)으로
  돌아오는지 반드시 확인**

## 2. 일반 인시던트 대응 프로세스 (PagerDuty Field Guide + Google SRE Book 근거)

두 자료가 공통으로 강조하는 순서: **완화(mitigate) 먼저, 근본 원인 규명은 그 다음.**
서비스를 빨리 정상으로 되돌리는 것과 "왜 그랬는지 알아내는 것"은 다른 작업이고, 인시던트
중에는 전자가 우선이다(롤백이 원인 조사보다 항상 먼저).

1. **알림 확인 즉시**: 위 1장의 해당 항목을 펼쳐서 확인 명령부터 실행(생각하지 말고 표대로)
2. **완화**: 롤백/재시작/방화벽 원복 등 "일단 정상으로 되돌리는" 조치를 먼저 취한다
3. **공유**: 팀 채널에 무슨 일이 있었는지 간단히 공유(다른 사람이 같은 걸 중복 조사하지 않게)
4. **근본 원인 조사는 완화 이후에**: 서비스가 정상화된 뒤 왜 그랬는지 조사
5. **기록**: 처리한 내용을 남긴다(나중에 포스트모템 쓸 재료가 됨 — 처리 당시엔 기억 안 나서
   못 쓰는 게 가장 흔한 실패 패턴)
