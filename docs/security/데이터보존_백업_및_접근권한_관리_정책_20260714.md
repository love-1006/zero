## 1. 데이터 보존(Retention) 정책

### 1-1. Prometheus (메트릭)

- **공식 근거**: Prometheus 공식 문서(Storage) — `--storage.tsdb.retention.time` 플래그로
  보존 기간을 설정하며, "이 플래그와 `storage.tsdb.retention.size`를 둘 다 지정하지 않으면
  보존 기간은 기본값 `15d`(15일)"이다. `--storage.tsdb.retention.size`는 기본 `0`(비활성화),
  즉 용량 기준 삭제는 별도로 켜지 않는 한 동작하지 않는다.
- **현재 설정**: `infra/monitoring/docker-compose.yml`의 prometheus 서비스 `command`에는
  `--storage.tsdb.retention.time`이 지정돼 있지 않음 → **기본값 15일이 그대로 적용 중**.
- **판단**: 지금 규모(VM 3~4대, 지표 개수 적음)에서는 15일 기본값으로도 디스크 부담이 크지
  않아 그대로 유지해도 무방. 다만 서비스 트래픽이 늘어나면(카디널리티 증가) 재검토 대상 —
  이미 학습자료 10절 "라벨 카디널리티 관리"에 언급된 리스크와 동일한 맥락.

### 1-2. Grafana Loki (로그) — ⚠️ 현재 무기한 보존 중 (실제 갭)

- **공식 근거**: Loki 공식 문서(Retention) — "기본적으로 `compactor.retention-enabled` 플래그는
  설정돼 있지 않아서, Loki로 보내진 로그는 영원히 산다(live forever)." 보존을 켜려면
  `compactor.retention_enabled: true`를 명시하고, `limits_config.retention_period`로
  전역 보존 기간을 지정해야 한다. 최소 설정 가능 단위는 24시간이며, 보존 기능을 쓰려면
  index period가 24h여야 한다는 전제조건도 있음.
- **현재 설정**: `docker-compose.yml` 주석에 명시된 대로 "이미지에 기본 번들된
  `/etc/loki/local-config.yaml` 그대로 사용" 중 — 즉 `retention_enabled`를 켠 적이 없어서
  **로그가 무기한 쌓이고 있는 상태**. 지금은 트래픽이 적어 체감이 안 되지만, 이건 방치하면
  모니터링 VM 디스크를 서서히 잠식하는 실제 리스크임.
- **권장 조치**: `limits_config.retention_period: 336h`(14일, Tempo와 동일 기준으로 통일)로
  설정하고 `compactor.retention_enabled: true` 활성화. 코드 변경이 필요한 항목이라 이 문서와는
  별개로 실제 반영 시점은 별도 작업으로 처리.

### 1-3. Grafana Tempo (트레이스)

- **공식 근거**: Tempo 공식 문서(Configuration - Compactor) — `compactor.compaction.block_retention`
  기본값은 `336h`(14일).
- **현재 설정**: `infra/monitoring/config/tempo.yaml`에 `block_retention: 48h`(2일)로 **기본값보다
  짧게** 명시적으로 설정돼 있음. 트레이스는 메트릭/로그보다 용량이 크고 지금은 디버깅 목적이
  강해서 짧게 잡은 것으로 보이며, 의도적 설정이라면 그대로 유지 가능. 다만 "왜 48h인지"에 대한
  근거가 기존 문서에 없었다면, 이 문서를 그 근거 기록으로 삼는다.

### 1-4. Harbor (이미지) — 이미 구현 완료

- 이미지 보존 정책은 이미 구축 초기에 확정·적용됨(오래된 이미지 자동 삭제). 별도 조치 불필요.

## 2. 백업(Backup) 정책

### 2-1. VM 레벨 백업 — Proxmox vzdump

- **공식 근거**: Proxmox VE 공식 문서(vzdump) —
  - 백업 모드 3가지: **Snapshot**(가장 짧은 다운타임, 약간의 비일관성 리스크 — guest agent가
    있으면 `guest-fsfreeze-freeze`/`thaw`로 일관성 보강), **Stop**(가장 높은 일관성, 짧은
    다운타임 발생), **Suspend**(호환성 목적으로만 존재, "Snapshot 모드 사용을 권장"한다고 명시).
  - 스케줄링은 `pvescheduler` 데몬이 담당하며, 호스트가 꺼져있어 놓친 작업을 자동 재실행하는
    "Repeat missed" 옵션도 있음.
  - 보존(prune)은 `prune-backups` 옵션으로 세분화: `keep-last`(최근 N개), `keep-daily`,
    `keep-weekly`, `keep-monthly`, `keep-yearly` 등. "각 옵션은 자신의 기간만 커버하고, 그
    상위 옵션이 이미 커버한 백업은 다시 세지 않는다"는 규칙이 있음. `protected` 표시된 백업은
    prune 대상에서 제외됨.
- **권장 설정** (harbor/DB/monitoring 3개 VM 공통):
  - 모드: **Snapshot** (공식 권장 + 다운타임 최소화, 팀 공용 서버라 중단 최소화가 중요)
  - 스케줄: 1일 1회(새벽 시간대, CI 빌드가 없는 시간)
  - 보존: `keep-daily 7, keep-weekly 4` (최근 1주는 매일 단위로, 그 이전 1개월은 주 단위로) —
    디스크 여유가 크지 않은 현재 자원 상황(Proxmox 호스트 31.25GB 중 이미 21.6GB 사용 중)을
    고려해 과도한 세대수는 피함.
- **현재 상태**: 실제 백업 작업(job) 설정 여부는 Proxmox 콘솔에서 별도 확인 필요 — 이 문서
  작성 시점 기준 설정 여부 미확인. **확인 후 미설정이면 최우선 조치 대상**(설정 자체가 코드
  변경이 아니라 Proxmox GUI/CLI 작업이라 이 문서 밖에서 처리, DB VM 포함 전체는 성애님 영역).

### 2-1-1. RPO/RTO 명시 (NIST SP 800-34 Rev.1 기준)

- **공식 정의**: NIST 공식 문서(SP 800-34 Rev.1) —
  - **RPO(Recovery Point Objective)**: *"장애 이후 데이터가 복구되어야 하는 시점(The point in
    time to which data must be recovered after an outage)"*
  - **RTO(Recovery Time Objective)**: *"정보시스템 구성요소가 조직의 임무에 부정적 영향을
    주기 전까지 복구 단계에 머물 수 있는 전체 시간"*
- **우리 환경에 적용**: 백업 주기를 "1일 1회"로만 써두면 RPO가 암묵적으로만 존재하는 셈이라,
  명시적으로 못 박는다 — **RPO ≈ 24시간**(최악의 경우 마지막 백업 이후 발생한 하루치 데이터
  유실 가능), **RTO 목표치는 아직 미측정**(실제로 백업에서 VM을 복구해본 적이 없어 소요시간을
  모름 — 아래 2-1-2 복구 테스트로 측정 예정).

### 2-1-2. 백업 복구 테스트(Restore Test) — 신규 추가, 실제 갭

- **왜 필요한가**: 백업이 "떠 있다"는 것과 "실제로 복구된다"는 것은 다른 문제다. 한 번도
  복구해본 적 없는 백업은 검증되지 않은 백업이다(백업 파일이 손상됐거나, 복구 절차 자체가
  누락된 경우 장애 시점에야 발견하게 됨).
- **권장 절차**: 분기 1회, 실제 백업 이미지로 **별도 테스트 VM을 복구**해서 부팅·서비스 기동
  여부까지 확인. 이때 측정된 소요시간이 실질적인 RTO 실측치가 된다.
- **현재 상태**: 미시행. 07/16 운영문서(Runbook) 작업 시 절차를 구체화할 예정.

### 2-1-3. 이중화/오프사이트 백업 — 3-2-1 규칙 (CISA 공식)

- **공식 근거**: CISA(미국 사이버보안·인프라 보안국) "Data Backup Options" 공식 문서 —
  *"To increase your chances of recovering lost or corrupted data, follow the 3-2-1 rule:"*
  - **3** — *"Keep 3 copies of any important file: 1 primary and 2 backups."* (중요 파일은
    원본 1개 + 백업 2개, 총 3벌 보관)
  - **2** — *"Keep the files on 2 different media types to protect against different types of
    hazards."* (서로 다른 재해에 대비해 2종의 서로 다른 매체에 보관)
  - **1** — *"Store 1 copy offsite (e.g., outside your home or business facility)."* (그 중
    1벌은 물리적으로 분리된 오프사이트에 보관)
- **우리 상황 대입**: 지금 Proxmox vzdump 백업이 **원본 VM과 같은 물리 호스트/스토리지에
  저장되는지, 별도 스토리지로 나가는지** 확인이 안 됨. 만약 같은 호스트라면 3-2-1 규칙 중
  "1(오프사이트)"과 "2(다른 매체)"를 위반하는 셈이라, 호스트 자체 장애(디스크 고장, 화재 등)
  시 원본과 백업이 동시에 소실된다. **성애님께 확인 요청 필요**(인프라 담당 영역) — 최소한
  백업본 1벌은 다른 물리 장치/스토리지로 분리하는 게 공식 권장.

### 2-2. 설정/코드 레벨 백업 — Git 버전관리

- **공식 근거**: Grafana 공식 문서(Provisioning) — "데이터소스와 대시보드를 버전관리 가능한
  파일로 정의할 수 있으며, 이는 GitOps를 더 자연스럽게 만든다(making GitOps more natural)."
- **현재 상태**: `infra/monitoring/dashboards/{alert-rules,zerocheck-overview}.json`을 API로
  주기적으로 export해서 git에 커밋하는 방식으로 이미 실천 중(오늘도 갱신함). 이건 공식 문서가
  권장하는 "버전관리되는 설정"이라는 목표를 달성하고 있음 — 다만 방향이 "git → Grafana"가
  아니라 "Grafana → git"이라는 차이가 있음(자세한 판단은 3절 참고).
- **비밀 관리**: 웹훅 URL 등 시크릿은 절대 이 백업 파일에 포함하지 않고 `harbor.credentials.local.md`(gitignore 처리)에만 기록 — 매 백업마다 `grep`으로 확인 후 커밋하는 절차를 유지.

## 3. 접근권한(Access Control) 관리 정책

### 3-1. VM SSH 접근

- 원칙: 비밀번호 로그인보다 **SSH 키 기반 인증 우선**(harbor/monitoring VM은 이미 로컬 키 등록
  완료 상태). 비밀번호는 팀 공용 규칙(내용은 `harbor.credentials.local.md` 참고 — gitignore
  처리된 파일이라 이 문서엔 평문 기재하지 않음) 유지하되 sudo 등 2차 인증 용도로만 사용.
- DB VM은 아직 이 환경 기준 Tailscale 미등록 상태(다른 팀원 계정 소유) — 접근이 필요해지면
  Share 요청 후 동일 원칙(SSH 키 등록) 적용.

### 3-1-1. 관리자 계정 다단계 인증(MFA) — 신규 추가

- GitHub/Grafana 모두 계정 자체에 2단계 인증(2FA/MFA)을 지원한다. 지금은 인원이 적어 단일
  계정 공유 구조라 우선순위가 낮았지만, 저장소 admin 권한을 지훈님도 부여받을 예정(팀장
  확인)이므로 **admin 권한을 갖는 모든 계정은 MFA 필수**로 정책화한다. GitHub는 조직 차원에서
  "Require two-factor authentication"을 강제하는 설정이 있어 팀장 권한으로 조직 전체에 적용
  가능 — GitHub 저장소 보안 설정 3건에 이어 팀장에게 추가 요청할 항목으로 남긴다.

### 3-2. Grafana 접근권한

- **공식 근거**: Grafana 공식 문서(Roles and Permissions) — 기본 조직 역할 3단계:
  - **Admin**: "조직의 모든 리소스(대시보드, 사용자, 팀 포함)에 접근 가능"
  - **Editor**: "대시보드/폴더/플레이리스트를 보고 편집 가능" (사용자 관리·데이터소스 관리는 불가)
  - **Viewer**: "대시보드/플레이리스트를 보고 데이터소스에 직접 쿼리 가능" (읽기 전용)
- **현재 상태 — ⚠️ 갭**: 지금은 `admin` 계정 하나를 팀이 공유하는 구조. 인원이 지훈님 혼자인
  현재는 문제가 없지만, **다른 팀원이 Grafana를 보게 되는 시점부터는 계정을 분리하고 역할을
  나눠야 함**(예: 알림 규칙을 직접 안 건드릴 팀원은 Viewer, SecOps/모니터링 담당만 Admin).
  지금 문서에 남겨두는 이유: 나중에 팀원 접근이 필요해질 때 "그때 가서 고민"이 아니라 이미
  정해둔 기준대로 바로 적용하기 위함.

### 3-3. GitHub 저장소 접근

- 브랜치 보호 규칙(Branch Protection) 적용은 학습자료 7절 "CI/CD 품질 개선 방안"에 이미
  제안돼 있음 — 이 문서에서는 접근권한 관점만 보강: main 브랜치 직접 push 금지 + PR 승인
  필수를 "프로세스적 접근권한 통제"로 정의하고, 실제 배포 게이팅(`if: github.event_name ==
  'push' && github.ref == 'refs/heads/main'`)을 "기술적 접근권한 통제"로 정의 — 두 층위가
  이미 갖춰져 있거나(기술적) 제안 상태(프로세스적)임을 명확히 구분해서 기록.
