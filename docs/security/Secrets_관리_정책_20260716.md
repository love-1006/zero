## 0. 왜 지금 쓰는가

과거 SonarQube 관리자 비밀번호가 공개 저장소 커밋에 평문으로 노출된 사고가 있었다(07/12,
`harbor.credentials.local.md` 도입 계기). 그 이후로는 "시크릿을 추적 파일에 평문으로 남기지
않는다"는 규칙만 지키고 있었는데, 이건 사고 재발 방지 수준의 임시 조치였지 정책은 아니었다.

07/15 밤~07/16 아침 사이 재헌님이 실제 백엔드 서비스 8개(admin/main/community/login/diet/
ingredients/product/recipe-service)를 만들면서 JWT 서명키, OAuth 클라이언트 시크릿, DB
비밀번호 같은 **진짜 시크릿**이 코드베이스에 등장하기 시작했다. "언젠가 필요할 정책"이 아니라
지금 당장 적용 대상이 있는 정책이 됐다.

## 1. 기본 원칙 (OWASP Secrets Management Cheat Sheet 근거)

OWASP 공식 치트시트의 핵심 원칙을 그대로 채택한다.

1. **시크릿은 코드에 하드코딩하지 않는다** — 코드와 설정(시크릿)을 분리한다.
2. **시크릿은 환경마다 다른 값을 쓴다** — 로컬 개발용과 운영용이 같은 값이면 안 된다.
3. **시크릿은 최소 권한으로 접근 가능해야 한다** — 그 시크릿이 실제로 필요한 서비스/사람만 볼 수 있어야 한다.
4. **시크릿은 주기적으로 로테이션한다** — 특히 유출 이력이 있거나 여러 사람이 공유하는 시크릿일수록.
5. **시크릿 유출은 커밋 시점에 막는 게 가장 싸다** — 유출되고 나서 지우는 것보다 push 자체를 막는 게 압도적으로 저렴하다(GitHub Push Protection의 설계 목적).

## 2. 계층별 관리 방식 (지금 실제로 쓰고 있는 방식 + 정식화)

| 계층 | 지금 하는 방식 | 정책으로 명문화 |
|---|---|---|
| **로컬 개발** | `.env.example`(실제 값 없이 키 이름만, git 추적) + `.env`(`.gitignore` 처리, 실제 값) | 이미 올바른 패턴(Twelve-Factor App "Config" 원칙 — 설정은 환경변수로, 코드와 분리). **유지.** 다만 `.env.example`에도 실제 인프라 주소(IP 등)는 적지 않고 `<POSTGRES_HOST>` 같은 자리표시자만 쓴다(2-1 참고). |
| **CI/CD (GitHub Actions)** | GitHub 암호화 Secrets(`SONAR_TOKEN`, `HARBOR_ROBOT_USER/TOKEN`, `SLACK_WEBHOOK_URL`) | 이미 GitHub 공식 권장 방식(Encrypted secrets, 저장 시 암호화 + 로그에 자동 마스킹). **유지.** 새 백엔드 서비스가 CI에서 실제 시크릿이 필요해지면(예: 배포 시 `JWT_SECRET` 주입) 같은 방식으로 추가. |
| **VM/운영 인프라** | `harbor.credentials.local.md`(gitignore, 로컬 파일 하나에 전부 기록) | 팀 규모(4인)에서는 과설계보다 실효성이 우선이라 지금 방식을 유지하되, **누가 이 파일을 갖고 있는지, 언제 갱신됐는지를 파일 상단에 명시**하도록 보완(3장 참고). 팀/서비스 규모가 커지면 Vault류 시크릿 매니저 도입 검토(지금은 과함). |
| **서비스 간 공유 시크릿** | 없음(지금까지 필요한 적이 없었음) | **신규 원칙 필요** — 2-2 참고. `JWT_SECRET`처럼 여러 서비스가 같은 값을 알아야 하는 시크릿이 처음 등장했다. |

### 2-1. 발견 사항 — `.env.example`에 실제 인프라 주소가 들어감

`backend/login-service/.env.example`(PR #19)에 다음과 같이 적혀 있다:

```
POSTGRES_HOST=211.46.52.151
POSTGRES_USER=여기에 입력
POSTGRES_PASSWORD=여기에 입력
```

비밀번호 자체는 비어있어 "시크릿 유출"은 아니지만, `.env.example`은 git으로 추적되는 공개
파일이라 **실제 서버 IP가 공개 저장소에 그대로 노출**된다. OWASP Secrets Management
Cheat Sheet는 시크릿뿐 아니라 "공격 표면을 넓히는 인프라 정보"도 최소 노출 원칙 대상으로
다룬다. `.env.example`은 "어떤 키가 필요한지"만 보여주면 충분하므로, 실제 값 대신
`POSTGRES_HOST=<db-host>` 같은 자리표시자로 바꾸는 걸 권장한다(재헌님과 공유 필요).

### 2-2. 발견 사항 — 서비스 간 공유 시크릿(`JWT_SECRET`)에 배포/로테이션 절차가 없음

`login-service`가 발급한 JWT를 `admin-service`가 검증하는 구조라, 두 서비스가
**정확히 같은 `JWT_SECRET` 값**을 가져야 한다(`admin-service/.env.example`의 주석에도
"Must be identical to login-service's JWT_SECRET"이라고 명시돼 있음). 지금은 두 파일 모두
`dev-secret-change-me`라는 동일한 하드코딩 문자열이 기본값으로 들어있다.

이건 우리 프로젝트에 **처음 등장한 유형의 문제**다 — 지금까지 시크릿은 전부 "서비스 하나가
혼자 쓰는 값"이었는데, 이제는 "여러 서비스가 같은 값을 알아야 하는" 시크릿이 생겼다.
- **운영 배포 시**: `JWT_SECRET`은 실제 랜덤값으로 한 번 생성해서, login-service/admin-service
  양쪽에 같은 GitHub Environment Secret(또는 VM의 `harbor.credentials.local.md` 같은 방식)으로
  동일하게 배포해야 한다. 두 서비스 저장소/설정에 각자 따로 값을 채워 넣다가 실수로 어긋나면
  admin 로그인이 통째로 깨지는 장애로 이어진다.
- **로테이션 시**: 한쪽만 바꾸면 안 되고 두 서비스에 동시 배포해야 한다는 걸 운영 절차에
  명시해야 한다.

## 3. 유출 탐지/차단 체계 현황 (실제 조회 결과)

| 항목 | 상태 | 확인 방법 |
|---|---|---|
| gitleaks pre-commit 훅 | ✅ 적용됨(`.pre-commit-config.yaml`, v8.30.0 고정) | 저장소 파일 직접 확인 |
| Dependabot alerts | ✅ **활성화 확인**(2026-07-16, API로 직접 재확인, 200 OK 빈 배열) | `gh api repos/love-1006/zero/dependabot/alerts` |
| Secret scanning / Push Protection | ⚠️ **확인 불가** — API가 404를 반환하는데, 이게 "기능이 꺼져있다"인지 "제 계정 권한(write, admin 아님)으로는 조회가 안 되는 것"인지 구분이 안 됨 | `gh api repos/love-1006/zero/secret-scanning/alerts` → 404. GitHub 공식 문서상 이 엔드포인트는 기능이 꺼져 있어도, 조회 권한이 없어도 404를 반환할 수 있어 이 응답만으로는 확정 불가 |

**조치**: admin 권한을 받으면(요청 발송된 상태) Settings → Code security에서 직접 화면으로
재확인하고, 꺼져 있으면 Push Protection까지 켜는 걸 최우선으로 한다 — 이번에 실제 시크릿이
포함된 PR이 올라오기 시작한 시점이라, 이 확인이 더 이상 미룰 수 있는 항목이 아니다.

## 4. 우선순위

| 항목 | 우선순위 | 비고 |
|---|---|---|
| Secret scanning/Push Protection 실제 활성화 여부 확인 | 🔴 필수 | admin 권한 확보 직후 1순위로 확인 |
| `JWT_SECRET` 운영 배포/로테이션 절차 팀 공유 | 🔴 필수 | 지금 하드코딩된 개발용 값을 그대로 배포하면 실제 보안사고, 재헌님과 직접 조율 필요 |
| `.env.example`의 실제 인프라 주소를 자리표시자로 교체 | 🟡 권장 | 시크릿 유출은 아니지만 공격 표면 노출 최소화 원칙 위반 |
| VM 자격증명 파일(`harbor.credentials.local.md`)에 관리자/갱신일 명시 | 🟡 권장 | 지금도 안전하게 관리되고 있으나 이력 추적성 보완 |
| 시크릿 매니저(Vault 등) 도입 | ⚪ 보류 | 현재 팀 규모(4인)·서비스 개수에서는 과설계, 서비스/인원이 늘어나면 재검토 |
