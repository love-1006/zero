## ⚠️ 이 문서로 "완성"되지 않는 부분 — 개발 착수 후 반드시 재작업 필요

이 문서는 **서비스가 뭘 하는지 모르는 상태에서 쓸 수 있는 데까지** 쓴 것이다. 아래 항목들은
문서를 아무리 다듬어도 채워지지 않는다 — **실제 서비스 스펙(데이터 종류, 인증 방식, 사용자
유형)이 나와야만 채울 수 있는, 원천적으로 다른 종류의 갭**이기 때문이다.

1. **A06 위협 모델링은 지금 일반론일 뿐이다** — "누가 무엇을 악용할 수 있는가"는 시스템이
   실제로 뭘 하는지 알아야 답할 수 있는 질문이다. ASVS에도 이걸 대신할 전용 조항이 없다는 걸
   확인했다(4장 참고). **개발 착수 직후, 실제 기능 목록을 놓고 위협 모델링 세션을 별도로
   진행해야 한다.**
2. **🔴🟡⚪ 우선순위는 "일반적인 웹앱이라면"이라는 가정으로 매긴 것이다** — 예를 들어 결제
   정보를 다루지 않는다면 A04(암호화) 일부 항목의 우선순위는 낮아져야 하고, 반대로 민감정보
   (건강/위치 등)를 다룬다면 올라가야 한다. **실제 데이터 분류가 나오면 우선순위 전체를
   재조정해야 한다.**
3. **국내 법적 요구사항(개인정보보호법 등)이 전혀 반영 안 됨** — OWASP/ASVS는 미국 중심
   표준이라 국내법 특유의 요구(수집 동의, 처리방침 고지, 국외 이전 제한 등)는 다루지 않는다.
   **어떤 개인정보를 수집하는지 확정되면 법무/컴플라이언스 검토를 별도로 추가해야 한다.**
4. **V9(토큰 무결성) 조항은 JWT 방식 인증을 전제로 한다** — 실제 인증 방식(세션 기반인지
   JWT인지, Keycloak을 나중에 쓸지)이 정해지지 않아서 이 전제가 맞는지 아직 모른다.

**이 4가지는 "문서 품질 문제"가 아니라 "정보가 아직 존재하지 않는 문제"다.** 지금 단계에서는
이 갭을 명시적으로 인지하고 넘어가는 것 자체가 맞는 방향이며, 개발 착수 시점에 이 문서를
반드시 재검토해야 한다는 걸 여기에 못 박아둔다.

> 이 문서는 "코드를 짤 때 이 10가지를 점검하라"는 체크리스트다. 이미 우리 CI/CD가 도구로
> 커버하는 항목(Trivy/gitleaks/SonarQube/Dependabot)은 명시했고, **도구가 못 잡고 개발자가
> 직접 신경 써야 하는 부분**을 중심으로 정리했다.
>
> ⚠️ **먼저 읽을 것**: 이 문서는 "서비스가 뭘 하는지 모르는 상태"에서 낼 수 있는 최대치까지
> 작성됐다. 위협 모델링(A06)·우선순위 재조정·국내 법적 요구사항 등 **서비스 스펙이 나와야만
> 채워지는 부분**은 문서 맨 아래 "이 문서로 완성되지 않는 부분" 섹션에 명시해뒀다 — 개발
> 착수 후 반드시 재검토할 것.

> **Top 10 vs ASVS — 이 문서의 근거 구조**: OWASP 공식상 Top 10은 "가장 심각한 리스크를
> 인지시키는" 목록이지 상세 검증 체크리스트가 아니다. OWASP가 실제 코드 검증용으로 제공하는
> 표준은 **ASVS(Application Security Verification Standard)**로, L1(기본)/L2(표준)/L3(고보증)
> 레벨로 수백 개 요구사항이 번호별로 나뉜다. **이 문서는 Top 10:2025를 뼈대로 삼되, 각
> 카테고리마다 실제 검증 가능한 ASVS 5.0.0 L1 조항(공식 저장소 CSV에서 직접 확인)을 근거로
> 붙였다** — "인지" 수준을 "검증" 수준으로 끌어올리기 위함.
>
> **ASVS 5.0 챕터 번호 주의**: ASVS 5.0에서 챕터가 재편됐다. 인증=V6, 세션=V7, 인가=V8,
> 인코딩/인젝션 방지=V1, 암호화=V11, 로깅=V16. (구버전 V2/V3 번호와 다름)
>
> **우선순위 기준**: 🔴 필수(데모/MVP 전 반드시) · 🟡 권장(여유 있으면 착수 시점부터) ·
> ⚪ 여지(스케일 커지면 재검토). 각 항목의 ASVS 레벨(L1/L2)도 병기 — **ASVS는 여러 항목을
> L2로 두므로, 우리 🔴(필수) 판정과 ASVS 레벨이 다를 수 있다**(우리는 프로젝트 맥락 기준).

## A01:2025 — Broken Access Control (접근 통제 실패)
> "가장 심각한 애플리케이션 보안 리스크"(OWASP 공식) — 40개 CWE 포함

- [ ] 🔴 기능 단위 접근 통제 — 명시적 권한 있는 사용자만 특정 기능 접근 가능
  - **ASVS V8.2.1 (L1)**: *"the application ensures that function-level access is restricted to consumers with explicit permissions"*
- [ ] 🔴 **객체 단위 권한 검증(IDOR/BOLA 방지)** — "내 데이터"인지 URL 파라미터만 믿지 말고 매번 서버에서 소유권 확인
  - **ASVS V8.2.2 (L1)**: *"data-specific access is restricted to consumers with explicit permissions to specific data items to mitigate insecure direct object reference (IDOR) and broken object level authorization (BOLA)"*
- [ ] 🔴 클라이언트 측(프론트엔드) 숨김만으로 접근 제어하지 않기 — 반드시 서버(신뢰 계층)에서 재검증
  - **ASVS V8.3.1 (L1)**: *"the application enforces authorization rules at a trusted service layer and doesn't rely on controls that an untrusted consumer could manipulate, such as client-side JavaScript"*
- [ ] 🟡 접근 통제 규칙을 문서화(권한별로 뭘 볼 수 있는지)
  - **ASVS V8.1.1 (L1)**: *"authorization documentation defines rules for restricting function-level and data-specific access based on consumer permissions and resource attributes"*
- [ ] 🟡 다른 권한(role)의 토큰/세션으로 테스트해서 실제로 막히는지 확인

## A02:2025 — Security Misconfiguration (보안 설정 오류)
> 2025년 2위로 격상 — 16개 CWE, 기본 설정/불완전한 구성/노출된 시스템 관련

- [ ] 🔴 운영 환경에서 debug 모드 비활성화(스택 트레이스 노출 금지)
- [x] 기본 계정/기본 비밀번호 전부 교체(우리 팀은 이미 `harbor.credentials.local.md` 규칙 있음)
- [ ] 🔴 서비스 간(백엔드↔DB 등) 인증에 기본 크리덴셜(admin/admin 등) 사용 금지
  - **ASVS V13.2.3 (L2)**: *"the credential being used by the consumer is not a default credential (e.g., root/root or admin/admin)"*
- [ ] 🟡 보안 헤더 설정(CSP, HSTS, X-Content-Type-Options 등)
- [ ] 🟡 Docker 이미지 설정 세부사항은 별도 문서(`Docker 이미지 보안 정의서`)로 관리

## A03:2025 — Software Supply Chain Failures (소프트웨어 공급망 실패)
> 우리가 이미 상당 부분 커버 중인 항목 — 5개 CWE, 의존성/빌드/배포 인프라 대상

- [x] 이미지 취약점 스캔 — **Trivy**(CRITICAL/HIGH 발견 시 파이프라인 중단)
- [x] 시크릿 유출 스캔 — **gitleaks**(전체 히스토리)
- [x] 의존성 CVE 알림 — **Dependabot alerts**(팀장 활성화 완료)
- [ ] 🔴 의존성 버전 고정(`requirements.txt`에 정확한 버전 명시, 범위 지정 지양)
- [ ] 🔴 취약점 있는 3rd party 컴포넌트에 대한 위험 기반 조치 기한을 문서화 + 그 기한을 넘긴 컴포넌트 없음을 유지
  - **ASVS V15.1.1 (L1)**: *"application documentation defines risk based remediation time frames for 3rd party component versions with vulnerabilities and for updating libraries in general"*
  - **ASVS V15.2.1 (L1)**: *"the application only contains components which have not breached the documented update and remediation time frames"* — Dependabot alerts(이미 활성화)로 발견 후 이 기한 안에 패치하는 프로세스 필요
- [ ] 🟡 SBOM 생성 및 신뢰할 수 있는 저장소에서만 컴포넌트 수급 확인
  - **ASVS V15.1.2 (L2)**: *"an inventory catalog, such as software bill of materials (SBOM), is maintained of all third-party libraries in use, including verifying that components come from pre-defined, trusted, and continually maintained repositories"* (Trivy `--format cyclonedx` 지원 확인됨, 학습자료 7장 참고)
- [ ] 🟡 GitHub Actions 서드파티 액션 SHA 고정(학습자료 보안 감사에서 지적된 항목)

## A04:2025 — Cryptographic Failures (암호화 실패)
> 32개 CWE — 암호화 구현 취약점, 민감정보 노출로 이어짐

- [ ] 🔴 안전한 암호/모드만 사용 — AES-GCM 등 승인된 것만, ECB 같은 취약 모드 금지
  - **ASVS V11.3.1 (L1)**: *"insecure block modes (e.g., ECB) and weak padding schemes (e.g., PKCS#1 v1.5) are not used"* / **V11.3.2 (L1)**: *"only approved ciphers and modes such as AES with GCM are used"*
- [ ] 🔴 안전한 해시 함수만 사용 — MD5 등 취약 해시 암호 용도 금지
  - **ASVS V11.4.1 (L1)**: *"only approved hash functions are used... Disallowed hash functions, such as MD5, must not be used for any cryptographic purpose"*
- [ ] 🔴 비밀번호는 전용 키 유도 함수(password hashing)로 저장 — 평문/단순 해시 금지
  - **ASVS V11.4.2 (L2)**: *"passwords are stored using an approved, computationally intensive, key derivation function"* — argon2/bcrypt/scrypt 계열(직접 구현 금지, 검증된 라이브러리)
- [ ] 🟡 모든 구간에 TLS 강제(내부망이라도 민감정보 전송 구간은 확인)
- [x] 하드코딩된 키/시크릿 금지 — 우리 팀 기존 원칙(SonarQube 비밀번호 유출 사고 이후 확립)과 동일 선상, gitleaks+Push Protection으로 이중 방어 중

## A05:2025 — Injection (인젝션)
> 38개 CWE — SQL Injection부터 XSS까지 가장 광범위하게 테스트되는 카테고리

- [ ] 🔴 DB 쿼리는 파라미터 바인딩/ORM 사용(SQLAlchemy 등), 문자열 조합으로 쿼리 생성 금지
  - **ASVS V1.2.4 (L1)**: *"data selection or database queries (SQL, HQL, NoSQL, Cypher) use parameterized queries, ORMs, entity frameworks, or are otherwise protected from SQL Injection"*
- [ ] 🔴 `subprocess`/셸 명령에 사용자 입력 직접 삽입 금지
  - **ASVS V1.2.5 (L1)**: *"protects against OS command injection and that operating system calls use parameterized OS queries or use contextual command line output encoding"*
- [ ] 🔴 출력값은 컨텍스트에 맞게 인코딩(XSS 방지) — HTML/JS/URL 각각 다른 인코딩
  - **ASVS V1.2.1 (L1)**: HTML 컨텍스트 출력 인코딩 / **V1.2.3 (L1)**: JS·JSON 출력 인코딩 / **V1.2.2 (L1)**: URL 동적 생성 시 인코딩 + 안전한 프로토콜만 허용
- [ ] 🔴 모든 외부 입력은 Pydantic 모델로 검증(이미 `Pydantic Settings` 패턴 쓰고 있어 확장 용이)
- [ ] 🟡 `eval`/`exec` 사용 금지 또는 최소화

## A06:2025 — Insecure Design (안전하지 않은 설계)
> 근본적 설계 결함 — 위협 모델링과 시큐어 설계 관행 강조

- [ ] 🟡 새 기능 개발 전 간단한 위협 모델링(누가 악용할 수 있는지, 신뢰 경계는 어디인지)
- [ ] 🔴 인증/인가 설계는 구현 전에 리뷰(Keycloak 미도입 상태이므로 자체 구현 부분 설계 리뷰가 더 중요)
- [ ] 🟡 API 응답은 필요한 필드만 반환(전체 데이터 객체를 그대로 반환하지 않기)
  - **ASVS V15.3.1 (L1)**: *"the application only returns the required subset of fields from a data object. For example, it should not return an entire data object, as some individual fields should not be accessible to users"*

> **정직한 한계**: ASVS는 "위협 모델링/설계"를 전용 챕터로 다루지 않는다. 위 V15.3.1은
> 방어적 코딩(과다 데이터 노출 방지) 조항이라 설계 단계라기보다 구현 단계에 가깝다 —
> 즉 A06의 핵심인 "설계 자체를 검증하는" ASVS L1 조항은 존재하지 않는다는 뜻이다.
> 이 카테고리는 ASVS보다 별도 위협 모델링 방법론(예: STRIDE)을 참고하는 게 더 정확하다.

## A07:2025 — Authentication Failures (인증 실패)
> 36개 CWE — 식별/인증 관련 취약점

- [ ] 🔴 로그인 rate limiting / anti-automation(크리덴셜 스터핑·무차별 대입 방지)
  - **ASVS V6.3.1 (L1)**: *"controls to prevent attacks such as credential stuffing and password brute force are implemented"*
- [ ] 🔴 기본 계정(root/admin/sa) 제거 또는 비활성화
  - **ASVS V6.3.2 (L1)**: *"default user accounts (e.g., 'root', 'admin', or 'sa') are not present in the application or are disabled"* (우리 팀 비밀번호 교체 규칙과 직접 연결)
- [ ] 🔴 비밀번호 정책 — 최소 8자(15자 권장), 변경 기능 제공, 변경 시 현재+새 비밀번호 요구
  - **ASVS V6.2.1 (L1)**: 최소 8자(15자 강력 권장) / **V6.2.3 (L1)**: 변경 시 현재+신규 비밀번호 요구 / **V6.2.5 (L1)**: 문자 구성 강제 규칙 금지(특수문자 강제 등 X) / **V6.2.7 (L1)**: 붙여넣기·비밀번호 관리자 허용
- [ ] 🔴 흔한/유출된 비밀번호 차단
  - **ASVS V6.2.4 (L1)**: *"checked against... at least, the top 3000 passwords"*
- [ ] 🔴 인증 세션 처리 — 로그인 시 새 세션 토큰 발급 + 기존 토큰 폐기
  - **ASVS V7.2.4 (L1)**: *"generates a new session token on user authentication... and terminates the current session token"*
- [ ] 🟡 비밀번호 재설정 — 초기 발급 코드는 랜덤·만료·1회용, 비밀번호 힌트/보안질문 금지

우선순위를 매겼다. 개발 착수 후 실제 ASVS L1 조항을 대조해 이 우선순위를 재검증해야 한다.
