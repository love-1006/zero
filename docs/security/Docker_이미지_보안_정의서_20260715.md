## 1. 체크리스트 (Docker 공식 근거)

### 1-1. 신뢰할 수 있는 베이스 이미지 사용
- **공식 근거**: *"The first step towards achieving a secure image is to choose the right
  base image."* Docker Official Images, Verified Publisher, Docker-Sponsored Open Source
  이미지를 우선 사용 — 이미 보안 검토를 거친 이미지들.
- [x] **충족** — `python:3.12-slim`은 Docker Official Image.

### 1-2. 최소 베이스 이미지 사용
- **공식 근거**: *"A smaller base image not only offers portability and fast downloads, but
  also shrinks the size of your image and minimizes the number of vulnerabilities introduced
  through the dependencies."*
- [x] **충족** — 이미 `-slim` 변형 사용 중(전체 이미지 대신).

### 1-3. 베이스 이미지 버전을 digest로 고정 🔴
- **공식 근거**: *"By pinning your images to a digest, you're guaranteed to always use the
  same image version, even if a publisher replaces the tag."*
- [ ] **갭 발견** — 지금 `FROM python:3.12-slim`은 태그일 뿐이라, 퍼블리셔가 같은 태그에
  다른 이미지를 올리면(드물지만 발생 가능) 우리도 모르게 다른 이미지가 빌드됨. digest
  고정(`FROM python:3.12-slim@sha256:...`)이 재현성·보안 면에서 더 안전.

### 1-4. Non-root 사용자로 실행 🔴 (가장 중요한 갭)
- **공식 근거**: *"If a service can run without privileges, use USER to change to a non-root
  user."* 그리고 *"Avoid installing or using sudo as it has unpredictable TTY and
  signal-forwarding behavior."*
- [ ] **갭 발견** — 지금 Dockerfile에 `USER` 지시어가 전혀 없어서 **컨테이너가 root로 실행
  중**. 컨테이너 탈출(escape) 취약점이 터지면 root 권한까지 그대로 넘어가는 구조.

### 1-5. Multi-stage build로 최종 이미지 최소화 — ✅ 적용 완료 (2026-07-16)
- **공식 근거**: *"Split your Dockerfile instructions into distinct stages to make sure that
  the resulting output only contains the files that are needed to run the application"*
  (공격 표면 축소 목적).
- `ci_sandbox`는 빌드 도구가 필요 없는 순수 Python 앱이라 급하지 않다고 판단했었지만,
  pip 캐시·wheel 빌드 산출물을 최종 이미지에서 배제하는 것 자체가 공식 문서가 명시한
  목적(공격 표면 축소)에 해당해 적용함. venv를 빌드 스테이지에 만들고 최종 스테이지로
  디렉터리만 복사하는, Docker 공식 Python 샘플의 표준 패턴 사용.
- **프론트엔드(빌드 도구 필요) 이미지는 여전히 개발 착수 후 재검토 항목** — 아직 이미지
  자체가 없음.

### 1-6. 불필요한 파일/패키지 제외
- **공식 근거**: `.dockerignore` 사용 + *"Avoid installing extra or unnecessary packages just
  because they might be nice to have."*
- [x] **충족** — `.dockerignore`에 `.git`/`.github`/`.claude`/캐시/문서 파일 이미 제외 중.

### 1-7. 이미지 정기 재빌드 🟡
- **공식 근거**: *"To keep your images up-to-date and secure, rebuild your images regularly
  with updated dependencies"*(`--pull` 플래그로 베이스 이미지 최신화).
- [ ] **확인 필요** — CI/CD의 `docker build` 단계가 `--pull` 옵션을 쓰는지 확인 필요(베이스
  이미지가 로컬 캐시에 있으면 오래된 버전을 계속 쓸 수 있음).

### 1-8. 컨테이너당 단일 역할
- **공식 근거**: *"Each container should have only one concern."*
- [x] **충족** — `ci_sandbox` 컨테이너는 uvicorn 프로세스 하나만 실행.

## 2. 런타임 보안 체크리스트 (OWASP Docker Security Cheat Sheet 기반) — 신규

**1장이 "이미지를 어떻게 빌드하는가"였다면, 이 장은 "그 이미지를 어떻게 실행하는가"다.**
Dockerfile을 아무리 잘 써도, 실행할 때 root 권한/무제한 리소스로 띄우면 소용없다.

### 2-1. 불필요한 권한(Linux capability) 전부 제거
- **공식 근거(OWASP RULE #3)**: *"Drop all capabilities `--cap-drop all` and then add only
  required ones."* 그리고 명확히 경고: *"Do not run containers with the `--privileged`
  flag!!!"*
- [ ] 🔴 **갭** — 지금 `docker-compose.yml`에 `cap_drop` 설정이 전혀 없음(기본값 = 여러
  capability가 살아있는 상태).

### 2-2. 권한 상승 방지
- **공식 근거(OWASP RULE #4)**: *"security-opt=no-new-privileges"*로 setuid/setgid 바이너리를
  통한 권한 상승을 차단.
- [ ] 🔴 **갭** — 미설정.

### 2-3. 읽기 전용 파일시스템
- **공식 근거(OWASP RULE #8)**: *"Run containers with a read-only filesystem using
  `--read-only` flag"*, 임시 쓰기가 필요하면 `--tmpfs`와 함께 사용.
- [ ] 🟡 **갭** — 미설정. `ci_sandbox` 코드를 직접 확인한 결과 파일 쓰기 코드가 없어서 적용
  리스크 낮음(단, Python 바이트코드 캐시 쓰기는 read-only 상태에서 조용히 실패하고 넘어감 —
  앱 동작에는 지장 없음).

### 2-4. 리소스 제한
- **공식 근거(OWASP RULE #7)**: 메모리/CPU/재시작 횟수/파일디스크립터/프로세스 수를 제한해서
  서비스 거부(DoS) 공격 완화.
- [ ] 🟡 **갭** — 미설정. 지금은 컨테이너 하나가 VM 리소스를 무제한으로 쓸 수 있는 상태.

### 2-5. Rootless 모드 (Docker 데몬 자체)
- **공식 근거(OWASP RULE #11)**: *"the Docker daemon and containers are running as an
  unprivileged user"* — 컨테이너 탈출이 일어나도 호스트에서 권한 상승이 안 되게.
- [ ] ⚪ **여지** — 이건 Dockerfile/Compose 레벨이 아니라 harbor VM의 Docker 데몬 설정 자체를
  바꿔야 하는 인프라 변경이라 영향 범위가 큼. 지금 당장 적용은 보류, 별도 검토 필요.

### 2-6. 공급망 보안 확장
- **공식 근거(OWASP RULE #13)**: 이미지 출처 문서화, SBOM, 이미지 서명, 신뢰할 수 있는
  레지스트리 사용.
- **NIST SP 800-190 4.1.5 보강**: *"discrete identification of each image by cryptographic
  signature"*, *"organizations should maintain a set of trusted images and registries and
  ensure that only images from this set are allowed to run"*.
- [x] Trivy 스캔(RULE #9)은 이미 적용 중. Harbor 자체가 "신뢰할 수 있는 사설 레지스트리"
  요구사항을 이미 충족(외부 공개 레지스트리 직접 사용 안 함). SBOM/이미지 서명은 코드단
  체크리스트 A03에 이미 동일 항목으로 반영돼 있어 중복 기재 안 함(상호 참조).

### 2-7. Docker 데몬 소켓 노출 금지
- **공식 근거(OWASP RULE #1)**: *"Do not enable tcp Docker daemon socket"*, 컨테이너에
  `/var/run/docker.sock`을 마운트하지 말 것 — 마운트하면 호스트에 대한 무제한 root 권한을
  컨테이너에 주는 것과 같음.
- [x] **충족** — `docker-compose.yml`에 `volumes:` 자체가 없어 소켓 마운트 여지가 없음.

### 2-8. 컨테이너 간 연결 통제 + 포트 바인딩
- **공식 근거(OWASP RULE #5)**: 기본 브리지 대신 커스텀 Docker 네트워크로 통신 범위를 명시
  적으로 통제. **RULE #5a**: 방화벽 우회 위험이 있으니 호스트에 포트를 열 때는
  `127.0.0.1:port:port`처럼 localhost로만 바인딩 권장.
- [x] **네트워크는 사실상 충족** — `docker compose`는 프로젝트별로 전용 네트워크를 자동
  생성하므로(Harbor의 별도 compose 스택과 자동 격리됨), 별도 커스텀 네트워크 명시는 선택
  사항.
- [ ] ⚪ **포트 바인딩은 의도적 예외** — 지금 `"8080:8000"`은 모든 인터페이스에 바인딩됨.
  RULE #5a대로 localhost 전용으로 바꾸면 monitoring VM의 Blackbox Exporter가 이 포트에
  접속할 수 없게 돼 오히려 우리 모니터링 구조가 깨짐. **네트워크 전체가 아니라 Tailscale로
  격리된 사설망이라는 걸 근거로 현재 바인딩을 의도적으로 유지**한다 — 자동 갭이 아니라
  검토 후 채택한 예외.

### 2-9. Linux 보안 모듈(LSM) 활용
- **공식 근거(OWASP RULE #6)**: seccomp/AppArmor/SELinux 프로파일 적용 + Falco 같은 행위
  기반 이상탐지 도구 추가 권장.
- [x] **기본 보호는 이미 적용 중** — Docker는 커스텀 프로파일을 지정 안 해도 **기본 seccomp
  프로파일을 자동 적용**한다(위험한 syscall 다수를 기본 차단). 커스텀 프로파일/Falco는
  ⚪ 여지 — 지금 규모에서는 과잉, 실제 서비스 트래픽 생기면 재검토.

### 2-10. Docker 데몬 로깅 레벨
- **공식 근거(OWASP RULE #10)**: `/etc/docker/daemon.json`의 로그 레벨을 기본값 `info`로
  유지.
- [ ] ⚪ **확인 필요, 인프라 레벨** — 이건 컨테이너가 아니라 harbor VM의 Docker 데몬 자체
  설정이라 이 문서(이미지/컨테이너 단위) 범위 밖. 인프라 점검 시 확인 항목으로만 남김.

### 2-11. Docker Secrets로 민감정보 관리
- **공식 근거(OWASP RULE #12)**: 비밀번호/토큰/SSH 키를 이미지나 커맨드에 직접 넣지 말고
  Docker Secrets로 관리.
- [x] **현재는 해당 없음** — `ci_sandbox`는 지금 DB 접속정보 등 시크릿을 전혀 안 씀(단순
  헬스체크 앱). **실제 백엔드 이미지가 DB/외부 API 자격증명을 쓰게 되면 이 규칙이 바로
  적용 대상이 됨** — 6장 "완성되지 않는 부분"에 명시.

### 2-12. 컨테이너 런타임 egress 트래픽 통제
- **NIST SP 800-190 4.4.2**: *"organizations should control the egress network traffic sent
  by containers"*, 앱 인지형 필터링 도구로 앱 특성에 맞는 규칙을 동적으로 생성 권장.
- [ ] ⚪ **여지** — 이건 애플리케이션 인지형 방화벽 같은 별도 네트워크 인프라 도구가 필요한
  고급 통제라, 지금 팀 규모/단계에서는 과함. 실제 서비스가 외부 API를 호출하기 시작하면
  재검토.

### 2-13. HEALTHCHECK 지시어 🔴 (CIS Benchmark, 신규 갭)
- **공식 근거(CIS Docker Benchmark v1.8.0, 4.6 "Ensure that HEALTHCHECK instructions have been added to container images" / 5.27 "Ensure that container health is checked at runtime")**: *"Add HEALTHCHECK instruction in your docker
  container images to perform the health check on running containers."* / *"If the container
  image does not have an HEALTHCHECK instruction defined, use `--health-cmd`."*
- [ ] **갭 발견** — 지금 Dockerfile에 `HEALTHCHECK` 지시어가 없음. 이미 CI/CD의 `deploy`
  job이 배포 후 `/health`를 curl로 확인하고 있지만, 그건 "배포 시점 1회성 체크"이고
  HEALTHCHECK는 **컨테이너가 살아있는 내내 Docker 데몬이 주기적으로 확인**하는 것이라
  다른 레이어. `docker ps`의 STATUS 컬럼에 healthy/unhealthy가 바로 보여서 운영 편의성도
  올라감.

### 2-14. Setuid/Setgid 권한 제거 🟡 (CIS Benchmark, 신규 갭)
- **공식 근거(CIS docker-4.8)**: *"Removing setuid and setgid permissions in the images would
  prevent privilege escalation attacks in the containers."*
- [ ] **갭 발견** — 베이스 이미지에 기본 포함된 일부 바이너리(예: `passwd`, `mount` 등)에
  setuid 비트가 남아있을 수 있음. non-root 사용자로 돌리고 있어 실질 위험은 낮지만,
  방어 심층화(defense in depth) 차원에서 제거 권장.

### 2-15. PID 제한(Fork Bomb 방지) 🟡 (CIS Benchmark, 신규 갭)
- **공식 근거(CIS Docker Benchmark v1.8.0, 5.29 "Ensure that the PIDs cgroup limit is used")**: *"Attackers could launch a fork bomb with a single command
  inside the container"* — PID 수를 제한해서 방지.
- [ ] **갭 발견** — `docker-compose.yml`에 `pids_limit` 미설정.

### 2-16. Docker Content Trust (이미지 서명) ⚪ (CIS Benchmark, 확인 필요)
- **공식 근거(CIS docker-4.5)**: *"Content trust is disabled by default. You should enable
  it."*(`DOCKER_CONTENT_TRUST=1`)
- [ ] **확인 필요, 적용 보류** — 이건 Harbor 레지스트리가 Notary(콘텐츠 신뢰) 기능을
  지원/활성화하고 있는지부터 확인해야 하는 인프라 의존적 항목. 코드단 체크리스트 A08에
  이미 있는 Sigstore/cosign(더 현대적인 서명 방식)과 목적은 같아서, 굳이 옛 Notary 방식을
  별도로 켤 필요는 없어 보임 — **Sigstore/cosign 채택으로 대체 예정, 중복 도입 안 함.**

### 2-17. 재시작 정책 — 검토 후 현행 유지 (CIS Benchmark 대조)
- **공식 근거(CIS Docker Benchmark v1.8.0, 5.15 "Ensure that the 'on-failure' container restart policy is set to '5'")**: *"Using the 'on-failure' restart policy and limit it to
  maximum of 5 restart attempts"* 권장.
- [x] **검토 완료, 현행 유지** — 지금 `restart: unless-stopped`는 CIS 권장(`on-failure:5`)과
  다르다. 다만 우리는 CI/CD의 `deploy` job이 헬스체크 실패 시 **이전 정상 이미지로 자동
  롤백**하는 별도 메커니즘을 이미 갖추고 있어서, `unless-stopped`(수동 중지 전까지 항상
  재시작)가 이 롤백 흐름과 더 잘 맞는다고 판단 — CIS 권장을 몰라서 놓친 게 아니라 우리
  상황에 맞게 의도적으로 다르게 선택한 것.

### 2-18. Docker 로그 로테이션 🟡 (Docker 공식, 2026-07-16 적용) — 체크리스트 Efficiency 항목
- **공식 근거(Docker 공식 문서, JSON File logging driver)**: `max-size`(기본값 무제한)와
  `max-file`(기본값 1, `max-size` 설정 시에만 동작) 옵션으로 로그 자동 로테이션을 켤 수 있다.
  기본값 그대로 두면 로그가 무제한으로 쌓여 호스트 디스크를 채울 수 있음.
- [x] **적용 완료** — `infra/ci-sandbox/docker-compose.yml`과 `infra/monitoring/
  docker-compose.yml`의 전 컨테이너에 `max-size: "10m"` / `max-file: "3"`(컨테이너당 최대
  30MB) 적용.

## 3. `ci_sandbox` 1차 시범 적용 — 실제 Dockerfile 대조

**현재 Dockerfile** (`Dockerfile`, 저장소 루트):
```dockerfile
FROM python:3.12-slim
WORKDIR /app
COPY ci_sandbox/requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY ci_sandbox/app ./app
EXPOSE 8000
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

**대조 결과**: 8개 체크리스트 중 5개 충족(1-1,1-2,1-6,1-8) + 1개 현재 해당없음(1-5) +
**2개 실제 갭**(1-3 digest 미고정, 1-4 non-root 미설정) + 1개 확인 필요(1-7).

**✅ 적용 완료** (2026-07-15, `Jhoon/secops/docker-image-hardening` 브랜치):
```dockerfile
FROM python:3.12-slim@sha256:c3d81d25b3154142b0b42eb1e61300024426268edeb5b5a26dd7ddf64d9daf28

# non-root 사용자로 실행 (Docker 공식 권장: USER로 전환)
RUN groupadd -r appuser && useradd -r -g appuser appuser

WORKDIR /app

COPY ci_sandbox/requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY ci_sandbox/app ./app

RUN chown -R appuser:appuser /app
USER appuser

EXPOSE 8000
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

digest는 Docker Hub Registry API(`registry-1.docker.io`)로 `python:3.12-slim`의 현재 manifest
list digest를 직접 조회해서 확보(`sha256:c3d81d25...`). CI/CD의 `docker build`에 `--pull`
플래그도 함께 추가해서 1-7 갭도 같이 닫음. 로컬에 Docker CLI가 없어 직접 빌드 테스트는
못 했고, **실제 검증은 self-hosted 러너의 CI 파이프라인에서 이뤄짐**(PR 열면 확인 가능).

**추가 확인(NIST SP 800-190 4.1.2)**: *"SSH and other remote administration tools...should
never be enabled within containers"* — `ci_sandbox` Dockerfile은 SSH 서버를 설치하지 않아
[x] **이미 충족**.

**추가 적용(2026-07-16, 멀티스테이지)**: 위 Dockerfile을 빌드 스테이지(venv에 의존성 설치)와
최종 스테이지(venv 디렉터리만 복사)로 분리했다. 1-5절 참고, 실제 파일은 저장소 루트
`Dockerfile` 참고.

## 4. `ci_sandbox` docker-compose.yml 대조 — 런타임 보안

**현재 설정** (`infra/ci-sandbox/docker-compose.yml`):
```yaml
services:
  ci-sandbox:
    image: harbor.hizero.local/dangdang/ci-sandbox:${IMAGE_TAG}
    container_name: ci-sandbox
    ports:
      - "8080:8000"
    restart: unless-stopped
```

**대조 결과**: 2장 6개 항목 중 런타임 보안 설정이 **전부 없음**(2-1,2-2,2-3,2-4 미설정).
Trivy 스캔(2-6)만 이미 적용 중.

**✅ 적용 완료** (capability 제거, 권한상승 방지, read-only, 리소스 제한 — Rootless 데몬
전환(2-5)만 인프라 영향이 커서 보류):
```yaml
services:
  ci-sandbox:
    image: harbor.hizero.local/dangdang/ci-sandbox:${IMAGE_TAG}
    container_name: ci-sandbox
    ports:
      - "8080:8000"
    restart: unless-stopped
    cap_drop:
      - ALL
    security_opt:
      - no-new-privileges:true
    read_only: true
    tmpfs:
      - /tmp
    environment:
      - PYTHONDONTWRITEBYTECODE=1
    mem_limit: 256m
    cpus: 0.5
```

**추가 적용(2026-07-16, 로그 로테이션, 2-18절)**: 위 설정에 `logging: {driver: json-file,
options: {max-size: "10m", max-file: "3"}}`를 추가했다. 실제 파일은
`infra/ci-sandbox/docker-compose.yml` 참고.

## 6. 이 문서로 완성되지 않는 부분

1. **프론트/백엔드 실제 이미지 구조를 모름** — 빌드 도구(webpack, 컴파일러 등)가 필요한지,
   필요하다면 멀티스테이지 설계를 어떻게 할지는 실제 이미지가 나와야 정할 수 있음.
2. **베이스 이미지 digest는 지금 고정해도 금방 stale해짐** — 보안 패치가 나올 때마다 갱신
   필요, 일회성 조치가 아니라 운영 프로세스로 만들어야 함(재빌드 주기와 연결).
3. **적정 리소스 제한값(mem_limit/cpus)은 추측치** — 실제 트래픽 패턴이 없어 256m/0.5 CPU는
   임의로 넉넉하게 잡은 값. 운영하며 실측 후 조정 필요.
4. **Rootless Docker 데몬 전환은 harbor VM 전체(Harbor 자체 포함)에 영향** — 이 이미지
   하나만의 문제가 아니라 인프라 결정이라 이 문서 범위 밖.
5. **Docker Secrets 관리 기준이 아직 없음** — 지금 `ci_sandbox`는 시크릿을 안 쓰지만, 실제
   백엔드 이미지는 DB 접속정보/외부 API 키를 다뤄야 함. 그때 이 문서에 Docker Secrets 또는
   동등한 방식(예: Kubernetes Secrets, 외부 시크릿 매니저)의 실제 적용 기준을 추가해야 함.
6. **Egress 트래픽 통제 기준이 없음(NIST 4.4.2)** — 실제 서비스가 외부 API를 호출하기
   시작하면, "이 컨테이너가 어디로 나가는 트래픽을 허용받는가"를 정의해야 함.
