# 당당 프론트엔드 3계층 연동 메모

이 문서는 현재 프론트엔드가 백엔드와 PostgreSQL 테스트 DB를 어떻게 바라보는지 정리한 메모입니다. 백엔드 소스와 DB 데이터는 수정하지 않았습니다.

## 요청 흐름

```text
브라우저
  -> Next.js 프론트엔드 (localhost:3001)
  -> /b 게이트웨이 (Next.js Route Handler)
  -> 기능별 FastAPI 서비스
  -> PostgreSQL 테스트 DB
```

- 브라우저에서는 백엔드 주소를 직접 사용하지 않고 `/b/...`만 호출합니다.
- `/b` 게이트웨이는 요청 경로에 따라 각 FastAPI 서비스로 전달합니다.
- 백엔드가 꺼졌거나 준비 중인 API라면 프론트엔드는 화면을 지우지 않고 준비된 목업 데이터를 보여줍니다.
- 인증 토큰이 있을 때만 사용자 전용 API를 호출합니다. 토큰이 없는 미리보기 상태에서는 DB 쓰기 요청을 보내지 않습니다.

## 서비스 매핑

| 프론트 경로 | 전달 대상 | 기본 포트 | 현재 용도 |
| --- | --- | ---: | --- |
| `/b/social-access/*`, `/b/user/*` | login | 8000 | OAuth, 최초 설정, 마이페이지 |
| `/b/home/*` | main | 8010 | 오늘 당류·칼로리, 건강 프로필, 추천 |
| `/b/community/*` | community | 8012 | 공지·커뮤니티·감미료 정보 |
| `/b/recipes/*` | recipe | 8014 | 레시피 목록·상세·대체 재료 |
| `/b/search`, `/b/product/*` | product | 8016 | 식품 검색·상세·개인화 설명 |
| `/b/tags/*` | ingredients | 8018 | 원재료·태그 |
| `/b/diet/*` | diet | 8020 | 식단 업로드·분석·캘린더 |
| `/b/admin/*` | admin | 8008 | 관리자 기능 |

명세서의 `/receipe` 표기는 실제 백엔드의 `/recipes`와 다릅니다. 프론트 게이트웨이는 두 표기를 모두 `/recipes`로 정규화합니다.

## 읽기 전용 DB 확인 결과

- `products`: 약 2,755건
- `product_tags`: 약 11,232건
- `tags`: 61건
- `recipes`: 약 1,697건
  - 만개의레시피 1,456건
  - 유튜브 241건
- `recipe_ingredients`: 약 13,574건
- `raw_ingredient_nutrients`: 약 3,704건
- 사용자, 건강 프로필, 식단 기록 관련 테이블은 현재 테스트 데이터가 비어 있습니다.

프론트 레시피 화면은 현재 요청에 맞춰 만개의레시피 출처만 합치며 유튜브 데이터는 노출하지 않습니다.

## 프론트에서 사용하는 실제 식품 분류

`가공식품`, `간편식·면류`, `견과·씨앗`, `곡물·시리얼`, `베이커리·간식`, `소스·조미`, `식물성 단백질`, `유제품`, `음료`, `잼·스프레드`, `특수영양식`

위 분류는 테스트 DB의 상품 태그 코드와 같은 순서와 이름을 사용합니다.

## 현재 연동과 폴백

| 화면/기능 | 실제 API 시도 | API가 없거나 실패할 때 |
| --- | --- | --- |
| 오늘 당류·칼로리 | `/b/home/user-sugar-calorie` | 로그인 미리보기 및 로컬 기록 |
| 마이페이지 | `/b/user/mypage`, `/b/home/health-profile` | 브라우저에 저장된 가입 정보 |
| 회원가입 목표 | `/b/user/firstset`, `/b/home/health-profile` | 저장 실패를 화면에서 안내하고 다음 단계로 이동하지 않음 |
| 레시피 목록·상세 | `/b/recipes`, `/b/recipes/{id}` | 만개의레시피 기반 로컬 데이터 |
| 식품 검색 | `/b/search` | DB에서 확인한 상품 기반 로컬 데이터 |
| 인기 순위 | `/b/home/rank/item` | 현재 준비된 랭킹 데이터 |
| 식단 기록 추가·삭제 | 대응 CRUD API가 아직 없음 | 날짜별 로컬 기록으로 동작 |

## 백엔드 연결 전에 확인할 사항

1. 명세의 레시피 경로 `/receipe`와 실제 경로 `/recipes` 중 공식 표기를 하나로 정해야 합니다. 프론트는 현재 둘 다 처리합니다.
2. 오늘 게이지가 main 서비스와 diet 서비스에 각각 있습니다. 목표값까지 제공하는 main 서비스 응답을 프론트 기준으로 사용하고 있습니다.
3. OAuth 시작 요청이 현재 응답하지 않으므로 login-service의 Redis 연결과 컨테이너 로그를 먼저 확인해야 합니다.
4. 로컬 OAuth 콜백과 `frontend_url`이 3000으로 설정돼 있다면, 실제 확인 전에 프론트 실행 주소 3001 또는 운영 도메인과 맞춰야 합니다. 백엔드 설정은 이번 작업에서 변경하지 않았습니다.
5. 식단 기록의 날짜별 추가·수정·삭제 API가 생기면 현재 로컬 저장 계층만 교체하면 됩니다. 화면 흐름은 유지할 수 있습니다.
6. 상세 배포·후속 작업은 `docs/PRODUCTION_HANDOFF.md`를 기준으로 확인합니다.
