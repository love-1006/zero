1. 안정성 & 고가용성 (Reliability & HA)

0 / 5 완료

  • 컨테이너 자동 복구 및 헬스 체크 구현

    모든 컨테이너에 `restart_policy: unless-stopped`가 적용되어 있고, 단순 프로세스 생존 상태 체크가 아닌 실제 포트 및 엔드포인트 응답(HTTP 200 등)을 검증하는 HEALTHCHECK가 설정되어 있습니다.

    Docker ComposeDocker Daemon
    가중치 4 (High)

  • Kafka 고가용성 설계 및 볼륨 마운트 영속화

    Kafka 브로커 장애 시 정합성 확보를 위해 Replication Factor가 최소 3(테스트/개발 2) 이상, `min.insync.replicas`는 2 이상, `acks=all`로 생산자가 전송하도록 지정되어 있으며, 데이터 디렉토리는 Host 볼륨에 마운트되어 영구 보존됩니다.

    KafkaDocker Volume
    가중치 5 (Critical)

  • 데이터베이스 백업 및 페일오버 자동화

    컨테이너화된 DB의 주기적 백업(pg_dump/mysqldump 등)이 스케줄링되어 별도 오브젝트 스토리지나 물리적 분리 드라이브에 안전하게 보관되며, 복구 테스트가 최소 연 1회 검증되었습니다.

    DatabaseShell Script
    가중치 4 (High)

  • 컨테이너간 DNS 및 명시적 종속성 설정

    웹/앱/DB 컨테이너 기동 순서 및 네트워크 통신을 보장하기 위해 `depends_on`에 `condition: service_healthy` 옵션이 적용되어 순차적 정상 가동 프로세스가 보장됩니다.

    Docker Compose
    가중치 3 (Medium)

  • 서킷 브레이커 및 재시도 로직 구현

    3-Tier 아키텍처 중 API 서버와 Kafka, DB 간 일시적 네트워크 순단 발생 시 무한 자원 소모를 방지하기 위한 지수 백오프(Exponential Backoff) 재시도 및 서킷 브레이커 로직이 애플리케이션 레벨에 적용되어 있습니다.

    Application
    가중치 4 (High)

    가중치 4 (High)

    GitHub ActionsTesting

3. 관찰 가능성 & 모니터링 (LGTM Stack)

0 / 5 완료

  • Prometheus 및 Loki 로그 보존 및 회전 정책

    Docker 컨테이너 로그 및 Loki에 저장되는 로그 데이터에 Log Rotation 정책(예: Docker daemon의 json-file max-size 설정)과 Loki Retention policy가 명시적으로 정립되어 디스크 폭발 장애를 방지합니다.

    LokiDocker Daemon
    가중치 5 (Critical)

  • Loki - Tempo - Grafana 분산 추적(Trace ID) 연동

    사용자 요청부터 DB 쿼리, Kafka 메시징까지 흐름을 추적할 수 있도록 애플리케이션 로그에 `trace_id`를 출력하고, Grafana에서 Loki 로그를 보던 중 원클릭으로 관련 Tempo 분산 트레이스로 이동 가능한 통합 파이프라인이 매핑되어 있습니다.

    LokiTempoGrafana
    가중치 4 (High)

  • Kafka 모니터링 (Consumer Lag 등) 수집

    Prometheus JMX Exporter 또는 Kafka Exporter를 통해 Kafka 브로커의 핵심 지표(Consumer Lag, Under-replicated Partitions, Active Controller Count)를 주기적으로 스크랩하고 Grafana 대시보드에 연동했습니다.

    Kafka ExporterPrometheus
    가중치 4 (High)

  • 자원 사용량 및 인프라 핵심 메트릭 대시보드 구성

    cAdvisor와 Node Exporter를 사용해 호스트 물리 장비뿐 아니라 개별 도커 컨테이너 레벨의 리소스 점유율(CPU, MEM, I/O)이 대시보드에서 일목요연하게 모니터링되고 있습니다.

    cAdvisorNode ExporterGrafana
    가중치 3 (Medium)

  • Grafana Alerting을 활용한 실시간 이상 상태 알림

    컨테이너 재부팅(Restarts), OOM 발생 우려(Mem 사용률 > 85%), Kafka Lag 과다 적체, 5xx 에러율 급증 시 슬랙(Slack), 디스코드 또는 이메일로 인스턴트 알림이 전송되도록 규칙이 활성화되어 있습니다.

    Grafana AlertingSlack
    가중치 5 (Critical)

4. 컨테이너 보안 (Container Security)

0 / 5 완료

  • 컨테이너의 Non-Root 실행 권한 제약

    Dockerfile 빌드 및 실행 시 `USER` 지정을 통해 루트 권한이 아닌 최소 권한의 시스템 사용자(예: appuser, node 등)로 애플리케이션을 구동합니다.

    DockerfileSecurity
    가중치 4 (High)

  • 경량화 베이스 이미지 채택 및 취약점 최소화

    빌드 및 구동 이미지로 불필요한 바이너리와 도구가 제거된 경량 이미지(Alpine, Distroless 또는 Slim 버전)를 사용하여 OS 수준 취약성 표면을 줄였습니다.

    DockerfileOptimization
    가중치 4 (High)

  • 망 분리 및 내부 전용 가상 네트워크 격리

    Web 프론트엔드 컨테이너만 외부에 포트를 노출하고 DB, Kafka, API 서버 등 백엔드 영역은 호스트 포트를 개방하지 않고 프라이빗 도커 네트워크(Docker Network) 내에서만 격리 통신합니다.

    Docker Network
    가중치 4 (High)

  • 컨테이너 런타임 호스트 읽기 전용(ReadOnly) 마운트

    컨테이너가 악성 코드에 오염되더라도 소스 영역의 불법적 변조를 완벽히 격리하기 위해 주요 어플리케이션 디렉토리를 `read_only: true` 또는 볼륨 마운트 시 `:ro` 옵션으로 사용 중입니다.

    Docker Volume
    가중치 3 (Medium)

  • 의존성 라이브러리 및 도커 보안 스캐닝 자동화

    주기적으로 Trivy, Dependabot, OWASP Dependency-Check 등을 활용해 Node_modules, PyPI 등의 서드파티 모듈과 OS 베이스 이미지 내 CVE 취약점을 모니터링하고 보완 조치하고 있습니다.

    SecOps
    가중치 3 (Medium)

    DockerfileOptimization
    가중치 4 (High)

    Kafka
    가중치 4 (High)

  • DB 인덱싱 및 캐싱 최적화

    Slow Query 로그를 기반으로 인프라 상의 비효율적인 DB 호출 쿼리에 적절한 인덱스를 적용하였고, 반복적인 읽기 요청은 Redis 등의 메모리 캐시나 내부 캐시 레이어를 두어 DB의 CPU 부담을 경감시킵니다.

    DatabaseQuery Tuning
    가중치 3 (Medium)

  • 빌드 캐시 무효화(Build Cache Invalidation) 순서 배치

    Dockerfile 작성 시 변경이 빈번히 일어나지 않는 `package.json`, `pom.xml`, `requirements.txt`와 같은 패키지 선언부를 먼저 COPY하고, 실제 소스 코드는 뒤쪽에 배치하여 Docker 빌드 캐시 낭비를 배제했습니다.

    Dockerfile
    가중치 3 (Medium)
